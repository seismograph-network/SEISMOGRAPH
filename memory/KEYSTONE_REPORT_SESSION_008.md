## 4. Interface conformance decision

**Requirement conflict resolved:** Tatiana's task specification requested
Redis Sets keyed by `model_tuple:metric_name`. The existing
`AgreementScorer` interface uses `model_tuple` only as the quorum scope
key. `ChangePointResult` has no `metric_name` field.

**Decision:** `RedisAgreementScorer` keys by `sg:quorum:{model_tuple}`
(matching the SESSION_007_HANDOVER acceptance criteria and the existing
interface). This preserves zero gateway code changes — the same
`scorer.ingest(cp) / scorer.promote_to_public_alert(mt) / scorer.clear(mt)`
call sequence works for both the in-memory and Redis backends.

**Known limitation KNOWN-LIMIT-001** (see Section 6): per-metric_name
Redis key granularity deferred to Phase 3, contingent on adding
`metric_name: str` to `ChangePointResult` and extending the scorer interface.

## 5. Known limitations

**KNOWN-LIMIT-001 — Metric-level quorum granularity not implemented**
Redis key is `sg:quorum:{model_tuple}` (one set per model, not per
model+metric). Two orgs with different drifting metrics on the same model
would satisfy quorum even if they are not drifting on the _same_ metric.
Mitigation: in Phase 2 MVP, a drifting model is the primary signal; metric
granularity is a refinement for Phase 3.
Resolution path: add `metric_name: str = ""` to `ChangePointResult`;
update `ingest()` to key by `sg:quorum:{model_tuple}:{metric_name}`;
extend `promote_to_public_alert(model_tuple, metric_name)` and
`clear(model_tuple, metric_name)` — requires Tatiana approval on
interface change.

**KNOWN-LIMIT-002 — No Redis connection health-check on startup**
`redis.Redis.from_url()` is lazy; connection errors surface only on the
first `sadd` call. A startup ping (`redis_client.ping()`) in lifespan
would surface misconfiguration immediately. Deferred for Phase 3 hardening.

**KNOWN-LIMIT-003 — Race window between promote and clear**
In a multi-node deployment, two gateway replicas could both call
`promote_to_public_alert()` before either calls `clear()`, producing
a duplicate `PublicDriftAlert`. Mitigation: QUORUM_MIN is 2, duplicate
alerts are idempotent in the weather API (status is already DRIFTING).
Phase 3 fix: Lua script `EVAL "if SCARD(key)>=quorum then DEL(key); return 1 else return 0"`.

## 7. Methodology note

The mock-first test approach (MagicMock for Redis client) is correct for
unit-level isolation but leaves the real Redis protocol unverified.
Suggested improvement: add one integration test in `tests/test_integration/`
that spins up a Redis container (via `pytest-docker` or a `docker run`
fixture) and exercises the full SADD → SCARD → DEL lifecycle with a real
server. This would catch protocol-level surprises (e.g. redis-py API
changes in v5+) that MagicMock cannot detect.
