## 4. Design decisions

**Sequential composition model (REQ-PRIV-011)**
DPAccountant uses the basic sequential composition theorem:
total epsilon = sum of per-flush epsilon values within the 24h window.
With FLUSH_EPSILON=2.0 and daily_budget=10.0, a probe may flush at most
5 times per day.  This is a conservative bound; Phase 3 may adopt the
Rényi DP advanced composition bound for tighter accounting, contingent
on Tatiana approval.

**Noop path does not deduct budget**
flush() with no pending results returns {"status": "noop"} immediately,
before the budget check.  This is correct: a no-op flush has zero
privacy cost.

**Budget check before DP noise application**
spend() is called before Aggregator.flush() applies Laplace noise.
If the budget is exceeded, no noise computation or HTTP activity occurs.
This eliminates the edge case where noise is applied to data that is
then discarded.

**`_accountant` injectable for testing**
DPAccountant is injected into ProbeSDK.__init__ via `_accountant` param,
consistent with the `_http_client` and `_key_manager` injection pattern.
Tests manipulate `accountant.window_start_time` directly for time-travel
(no `_now_fn` closure needed).

## 5. Known limitations

**KNOWN-LIMIT-004 — Wall-clock window; hibernation / clock-skew caveat**
reset_if_needed() uses datetime.now(timezone.utc) for window comparison.
System hibernation or NTP corrections may shorten the effective window.
Impact is under-reset (probe flushes less than expected), not over-reset,
so privacy guarantee is conserved.  Acceptable for Phase 2 MVP.

**KNOWN-LIMIT-005 — No persistent budget across process restarts**
DPAccountant is in-process; a gateway restart resets current_spend to 0.
A malicious or misconfigured probe operator could restart the probe
process to bypass the daily budget.  Phase 3 mitigation: persist
(current_spend, window_start_time) in Redis alongside the quorum state.

**KNOWN-LIMIT-006 — Per-flush budget cost not per-model-tuple**
flush() deducts FLUSH_EPSILON once per call regardless of how many
model_tuples have pending results.  A probe with 3 pending model_tuples
in one flush() call is charged 2.0 epsilon total, not 3 * 2.0.
This is a deliberate choice to keep the budget predictable for callers,
but it means high-cardinality probes may under-account privacy cost.
Flagged for Phase 3 review.

## 7. Methodology note

DP6 (24h reset test) uses direct time-travel via `window_start_time`
attribute mutation rather than a `_now_fn` injection.  This is pragmatic
but couples the test to the implementation's datetime field name.
Suggested improvement: expose a `_now()` override hook on DPAccountant
(analogous to `_key_manager` in ProbeSDK) so tests can inject a
deterministic clock without touching internal state.  This would also
enable property-based testing of reset boundary conditions.
